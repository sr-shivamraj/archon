🗺️ The Navigation Analogy

Imagine driving to a city you've never visited. You could ask a different stranger for directions at every intersection - each one knows their neighborhood, gives you their best answer, and the moment you turn the corner they forget you exist. You might eventually arrive, but you've repeated yourself a dozen times and you're starting from scratch at every turn.

Or you could use GPS. One pre-loaded map of the entire city. Context that persists from the first turn to the last. A system that knows where you've been, where you're going, and which route fits your constraints. Same destination - completely different journey.

Bare Claude is the stranger at the intersection. R+ Archon is the GPS. Both know the roads. Only one remembers the route.

The deeper principle comes from graph theory. In an unstructured Claude workflow, every conversation is an isolated node - no edges, no memory, no connection to what came before. In R+ Archon, sessions are nodes in a directed acyclic graph - knowledge, decisions, and context are edges that persist and accumulate. Every session you run adds to the graph. The Problem: AI Amnesia Meets Enterprise Complexity

The Rewards+ Platform is a multi-tenant loyalty marketplace powered by Capillary CRM. It has a BFF gateway (Java/Spring Boot/GraphQL), a custom API layer called Neo Extension (low-code dataflows with MongoDB), a React/MUI frontend, and integrations with half a dozen external systems per client.

Every time I started a new Claude conversation, I had to re-explain the entire architecture. The BFF uses SPQR for GraphQL, not Spring GraphQL. Neo Extension uses Basic Auth generaly for BFF integration. The UI must use MUI components exclusively - never raw HTML. Points drift tolerance is 0.01%. And a hundred other details that shape every design decision.

A good SDD is the single document a new engineer can read to understand the entire solution. If they have to ask "how does X work?" after reading it, the SDD is incomplete. The same standard should apply to what the AI knows about your platform.

The SDDs that Claude generated were surface-level - vague data flows, missing API contracts, no field mapping tables, generic error handling. They'd say "the user data is fetched and stored in Capillary" instead of showing the exact API endpoint, request payload, response shape, and error codes.

The fix wasn't better prompting. It was building a structured knowledge system that gives Claude deep, persistent understanding of the platform - and an agent architecture that channels that knowledge into production-quality outputs.

Token Optimization: The 5% Rule

Here's the engineering problem: Claude has a ~200K token context window. The full knowledge base is ~130K tokens. Loading everything would consume 65% of the context, leaving only 35% for actual reasoning and output. That produces shallow, truncated results.

The solution is a three-tier loading system:

Registry → Selector → Context-Load Token Architecture

  1. REGISTRY.md (~500 tokens) - A lightweight index listing every file with a one-line description and token cost. Claude reads this first to know what exists without loading anything.
  2. SELECTOR.md (~300 tokens) - A task-matching table: "If your task is X, load profile Y." Maps 26 common tasks to their optimal file sets.
  3. LOAD-*.md profiles (~200 tokens each) - Pre-computed "shopping lists" capping at 8-12K tokens. "Add GraphQL query" loads 4 specific files. "Write SDD" loads 4 different files.
  4. Fragments (~50-80 tokens each) - When you just need one code pattern (a SPQR annotation, a WebClient call), load a fragment instead of a 1,500-token skill file. That's a 95% token savings per pattern.

Result: Knowledge consumption drops from 65% → 5% of the context window. The remaining 80% is available for deep reasoning, long document generation, and multi-agent analysis.

Platform isolation: Declare your platform at session start (Platform: rewards-plus / capillary-core / generic) and the SELECTOR filters out irrelevant knowledge files automatically - no manual pruning needed. A non-Rewards+ engagement won't load Rewards+-specific types or patterns.

Architecture Debate System

For major design decisions, 5 debate agents evaluate the architecture from different perspectives - then a synthesizer produces the final recommendation:

🏛️

Conservative

"Do we really need this complexity? Could a simpler architecture achieve the same goals?"

📈

Scalability

"What breaks when we get 10x more traffic? Where are the bottlenecks?"

💰

Cost Efficiency

"Are we spending $10 to solve a $1 problem? Could a simpler solution work?"

🛡️

Stability

"What happens when this breaks at 3 AM? Is there a runbook? A fallback?"

The Decision Synthesizer reads all 5 evaluations, identifies consensus and conflicts, weighs tradeoffs against project context, and produces a final recommendation with a formal Architecture Decision Record (ADR).

18 Architecture Patterns - Including Resilience

Beyond structural patterns (BFF, Strangler, CQRS, DDD, Hexagonal, Modular Monolith), R+ Archon includes four resilience patterns that are critical for a multi-tenant platform calling external APIs:

⚡

Circuit Breaker

When Capillary is down, fail fast and serve from Redis cache instead of letting timeouts cascade. Includes Resilience4j code patterns with per-dependency fallbacks.

🔄

Retry with Backoff

Exponential backoff with jitter for transient failures. Includes a decision matrix: retry 5xx and timeouts, never retry 400s. Code for both BFF Reactor and Python batch scripts.

🚢

Bulkhead

Isolate failures per tenant and per dependency with separate connection pools. One bad tenant's traffic spike doesn't exhaust the pool for everyone else.

📤

Transactional Outbox

Guarantee event delivery by writing to a local outbox collection in the same Neo dataflow as the business write. A separate scheduled flow publishes to Kafka.

Quality Gates: SDD Completeness + Security Review

Two machine-checkable checklists ensure output quality before client sign-off. The SDD Completeness Checker verifies every feature section has API samples, field mapping tables, business rules, and error handling. The Security Review Checker walks through OWASP Top 10, secrets management, tenant isolation, and data privacy - every checkbox must pass.

Workspace Chunking: How It Handles 80-Page Documents

An 80-page SDD is roughly 20,000 tokens of output. Claude's context window can fill up mid-generation, potentially losing everything. The workspace chunking system prevents this:

  1. Break output into 2K-token chunks - each chunk is a self-contained section of the document, written to a separate file in _workspace/{task}/chunks/.
  2. Track progress in _status.json - records which chunks are complete, which section comes next, and any intermediate state.
  3. If context fills up - Claude saves progress, you start a new session, it reads _status.json + existing chunks, and continues from the last checkpoint.
  4. Assemble final document - when all chunks are done, they're concatenated into the final deliverable.

This means a 50-page SDD generation that would have crashed mid-way now survives context limits and can be resumed across multiple sessions.

Decision Checkpoints: The Guardrail That Changes Everything

The single most important feature in R+ Archon isn't the agents or the knowledge base - it's the checkpoint system that prevents the AI from silently making wrong assumptions. Without it, you could get a 50-page SDD built on the wrong architecture. With it, every significant decision is confirmed before execution.

There are 9 checkpoint types. Each fires at a specific point in the workflow:

CheckpointFires WhenWhat You Confirm
🏗️ ArchitectureSystem design beginsWhich systems, what pattern, data flow direction
📋 ContractAPI design beginsEndpoint path, method, request/response shape, fields
🔒 SecurityAuth model chosenOAuth vs API key vs SAML, PII handling, secret storage
💰 Business LogicLoyalty rules designedEarn ratios, tier thresholds, expiry model, milestones
🔀 MigrationMigration plannedStrategy (phased/big-bang/strangler), cutover window, rollback criteria

⏱️ EstimationEffort estimatedBreakdown, assumptions, risks - before it goes to stakeholders

Different commands trigger different checkpoints. An SDD generation gets 5 checkpoints (product selection, architecture, security, business logic, contracts). A dev spec gets 2 (contract, data model). A quick API question gets none. The system is proportional - it doesn't slow you down for simple tasks, but it catches wrong assumptions on big ones.

The rule is simple: never assume, always confirm. If a decision shapes the output in a way that would be costly to undo, the system pauses and asks first.

And if you're in a hurry? Say "skip checkpoints" and Archon proceeds without pausing - but it notes the assumptions it made, so you can review them in the final output.

Self-Correcting Knowledge

The library includes two self-correction mechanisms:

Validator files (_validators/) define rules for checking library integrity - broken cross-references, orphaned files, missing minimum content depth. After any update, these can be run to verify the library is consistent.

[INFERRED] tags appear throughout Capillary-specific content. These mark sections written from general Capillary knowledge that haven't been verified against the latest API docs. When a developer encounters an [INFERRED] section that's wrong, UPDATE_LEARNING corrects it - the fix persists for all future sessions and all team members.

Staleness detection prevents [INFERRED] sections from silently aging into wrong answers. Every knowledge file carries staleness_risk: high|medium|low and a last_verified date in its frontmatter. REGISTRY.md checks these at session start and flags any file overdue for re-verification as [VERIFY]. High-risk files (Capillary APIs) flag after 90 days. Medium-risk (BFF/Neo) after 180 days. Low-risk (methodology) after 365 days. Stale facts surface before they reach a production design.

The Critical Update Protocol prevents accidental corruption. Any write to memory files, architecture patterns, or knowledge types requires explicit confirmation. Claude shows exactly what will change and waits for approval before proceeding.

Most memory and skill plugins sync to cloud infrastructure. That means your codebase snippets, client decisions, architecture notes, and proprietary context leave your machine. For personal projects - fine. For enterprise work with client data, NDAs, or compliance requirements - this is a hard stop. Always ask: where is my context stored, who can access it, and what is the retention policy?

The Other Challenges

  • Fragmentation - Five plugins running in parallel with no shared context between them. Plugin A does not know what Plugin B learned. You end up with siloed capabilities, not a coherent system.
  • Generic outputs - A UI/UX plugin gives you generic best practices. It does not know your stack, your constraints, or your team's conventions. Every output needs manual correction for your actual context.
  • Vendor dependency - Free or paid, tools shut down, change terms, or get abandoned. Your accumulated context and workflows are at risk if they live on someone else's infrastructure.
  • Black-box retrieval - Semantic memory retrieval is probabilistic. You do not know what was surfaced and what was silently dropped. This is fine for casual use - dangerous when accuracy matters.
  • No workflow layer - A plugin gives you a capability, not a workflow. It cannot orchestrate: load the right context, apply the right template, write to the right output folder, and close with a memory update. That orchestration is what a structured local setup provides.
  • Token budget blindness - Most plugins have no concept of how much context they are consuming. Unbounded memory injection in a 200K window degrades output quality on the actual task.

Enterprise Why a Local Setup Is Non-Negotiable as the Foundation

Plugins are additive. A local setup is foundational. The difference: your local .claude/ library keeps all context on your machine, tracked in git, with no third-party servers in the chain. It is domain-specific to your actual stack - not a generic tool trying to cover every use case. It self-heals from real usage. And it is one coherent system where every component knows about every other. That is not something you can assemble from plugins alone.

Use plugins to extend capabilities. Use your local library as the foundation they extend.

Best Practices: The Short List

Regardless of your role or setup, these habits compound over time.

1

Always Start Fresh

New task = new session. Do not carry stale context from a previous conversation into a new problem. Old threads pollute reasoning. A clean session with the right context loads is faster and more accurate than a long running one.

2

Commit Small and Often

Commit after each meaningful output - a generated doc, a decision recorded, a spec written. Git is your audit trail. Small commits make it trivial to roll back a bad output without losing everything around it.

3

Close Every Session Properly

A session that ends without saving its decisions is a session that never happened. Run your close command (END_SESSION or equivalent) before you close the tab. Decisions not saved are decisions lost - and you will re-make them in the next session at full cost.

4

.claudeignore for Coders

Treat .claudeignore like .gitignore. Keep node_modules, build artifacts, .env files, credentials, and large generated files out of Claude's context. Irrelevant files waste tokens and dilute the signal. Set it up once, and every session is cleaner.

5

One Task Per Session

Mixing unrelated tasks in a single session degrades output quality on both. A session working on a database migration should not also be reviewing UI copy. Focused context produces focused output. When the task shifts, open a new session with the right context profile loaded.

The Bottom Line

The ecosystem is rich and growing fast. Plugins are useful - some are genuinely excellent. But no plugin replaces a well-structured local setup built for your actual domain, your actual stack, and your actual team. Build the foundation first. Add plugins where they extend - not replace - that foundation. And before any plugin touches enterprise work: audit where data goes, check vendor terms, and test against your security policy.

Your local setup is the moat. Plugins are the tools inside it.
